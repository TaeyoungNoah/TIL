### 검증 헤더와 조건부 요청 1

- 캐시 유효 시간이 초과해서 서버에 다시 요청하면 다음 두가지 상황 가능

> 1. 서버에서 기존 데이터를 변경함
> 2. 서버에서 기존 데이터를 변경하지 않음



**[캐시 만료후에도 서버에서 데이터를 변경하지 않음]**

- 생각해보면 데이터를 전송하는 대신에 저장해 두었던 캐시를 재사용 할 수 있다.

- 캐시 유효 시간이 초과해도, 서버의 데이터가 갱신되지 않으면

- 클라이언트는 서버가 보낸 응답 헤더 정보로 캐시의 메타 정보를 갱신
- 클라이언트는 캐시에 저장되어 있는 데이터 재활용
- 결과적으로 네트워크 다운로드가 발생하지만, 용량이 적은 헤더 정보만 다운로드
- **매우 실용적인 해결책**

### 검증 헤더와 조건부 요청 2

- 검증 헤더

> - 캐시 데이터와 서버 데이터가 같은지 검증하는 데이터
> - Last-Modified, ETag

- 조건부 요청 헤더

> - 검증 헤더로 조건에 따른 분기
> - If-Modified-Since: Last-Modified 사용
> - If-None-Match: ETag 사용
> - 조건이 만족하면 200 OK
> - 조건이 만족하지 않으면 304 Not Modified





#### 예시

> If-Modified-Since: 이후에 데이터가 수정되었으면?

- **데이터 미변경 예시**

> - 캐시: 2020년 11월 10일 10:00:00 vs 서버: 2020년 11월 10일 10:00:00
>
> - **304 Not Modified**, 헤더 데이터만 전송(BODY 미포함)
>
> - 전송 용량 0.1M (헤더 0.1M, 바디 1.0M)

- **데이터 변경 예시**

> - 캐시: 2020년 11월 10일 10:00:00 vs 서버: 2020년 11월 10일 **11:00:00**
> - 200 OK, 모든 데이터 전송 (BODY 포함)
> - 전송 용량 1.1M (헤더 0.1M, 바디 1.0M)





#### Last-Modified, If-Modified-Since 단점

- 1초 미만(0.x초) 단위로 캐시 조정이 불가능
- 날짜 기반의 로직 사용
- 데이터를 수정해서 날짜가 다르지만, 같은 데이터를 수정해서 데이터 결과가 똑같은 경우

> 예를 들어 서버가 a에서 b로 수정을 했다가 다시 a로 수정을 함 / 
>
> 이때 수정 날짜가 다르지만 사실상 같은 데이터를 웹에서 요청하고 있는 것임 /
>
> 하지만 이런 경우에도 수정 날짜가 다르므로 모든 데이터를 다 전송함

- 서버에서 별도의 캐시 로직을 관리하고 싶은 경우

> - ex) 스페이스나 주석처럼 크게 영향이 없는 변경에서 캐시를 유지하고 싶은 경우

- 위 두가지 문제를 해결하기 위해 서버에서 캐시를 제어할 수 있음 / 그 방법이 ETag, If-None-Match





#### ETag, If-None-Match

- ETag(Entity Tag)
- 캐시용 데이터에 임의의 고유한 버전 이름을 달아둠

> - ex) ETag: "v1.0", ETag: "a2jiodwjekjl3"
> - 파일을 Hash 알고리즘에 넣어서 그 결과를 받을 수 있는데 Hash는 파일이 동일하면 같은 값이 나옴
>
> > 이러면 위에서 a에서 b로 바꿨다가 다시 a로 바꾼 경우 똑같은 Hash값이 나옴

- 데이터가 변경되면 이 이름을 바꾸어서 변경함 (Hash를 다시 생성)

> - ex) ETag: "aaaaa" → ETag: "bbbbb"

- **진짜 단순하게  ETag만 보내서 같으면 유지, 다르면 다시받기**

### 캐시와 조건부 요청 헤더

#### 캐시 제어 헤더

- Cache-Control: 캐시 제어
- Pragma: 캐시 제어(하위 호환)
- Exprires: 캐시 유효 기간(하위 호환)



#### Cache-Control

> 캐시 지시어(directives)

- Cache-Control: max-age

> - 캐시 유효 시간, 초 단위

- Cache-Control: no-cache

> - 데이터는 캐시해도 되지만, 항상 원(origin) 서버에 검증하고 사용

- Cache-Control: no-store

> - 데이터에 민감한 정보가 있으므로 저장하면 안됨
>
>   (메모리에서 사용하고 최대한 빨리 삭제)



#### Pragma

> 캐시 제어(하위 호환)

- Pragma: no-cache
- HTTP 1.0 하위 호환 (지금은 거의 사용x , 가끔 필요시 사용)



#### Expires

> 캐시 만료일 지정 (하위 호환)

- expires: Mon, 01 Jan 1990 00:00:00 GMT



- 캐시 만료일을 정확한 날짜로 지정
- HTTP 1.0부터 사용해왔음
- **지금은 더 유연한 Cache-Control: max-age 권장**
- Cache-Control: max-age와 함께 사용하면 Expire는 무시



#### 검증 헤더와 조건부 요청 헤더

- **검증 헤더(Validator)**

> - **ETag**: "v10", **ETag**: "asid93jkrh2l"
> - **Last-Modified**: Thu, 04 Jun 2020 07:19:24 GMT

- **조건부 요청 헤더**

> - **If-Match**, **If-None-Match**: ETag 값 사용
>
> - **If-Modified-Since**, **If-Unmodified-Since**: Last-Modified 값 사용

### 캐시 무효화

#### Cache-Control

> 확실한 캐시 무효화 응답

- Cache-Control: no-cache, no-store, must-revalidate

- Pragma: no-cache

> - HTTP 1.0 하위호환

> 캐시를 안해도 웹 브라우저에서 유도리로 캐시를 만들어버릴 수 있음
>
> 그렇기 때문에 캐시를 사용하고싶지 않을 때  위 문구를 **모두 이용**



#### Cache-Control

> 캐시 지시어(directives) - 확실한 캐시 무효화 

- Cache-Control: no-cache

> - 데이터는 캐시해도 되지만, **항상 원 서버에 검증**하고 사용 (이름에 주의)

- Cache-Control: no-store

> - 데이터에 민감한 정보가 있으므로 저장하면 안됨
>
>   (메모리에서 사용하고 최대한 빨리 삭제)

- Cache-Control: must-revalidate

> - 캐시 만료 후 최초 조회시 **원 서버에 검증**해야함
>
> > Q: Cache-Control: no-cache는 항상 원서버에 검증하는데 굳이 이걸 왜 또 써야하지??
> >
> > A: 강의자료 참고
>
> - 원 서버 접근 실패시 반드시 오류가 발생해야함 - 504(Gateway Timeout)
> - must-revalidate는 캐시 유효 시간이라면 캐시를 사용함

- Pragma: no-cache

> HTTP 1.0 하위호환
